# Manager Card Tint Design

- 适用范围：TabVault manager 页面中的链接卡片视觉层次增强
- 最后更新时间：2026-05-25
- 关联文件：
  - `src/ui/manager/ManagerTabGrid.tsx`
  - `src/ui/shared/app-shell.css`
  - `src/types/session.ts`
  - `docs/onetab-like-extension-prd.md`
  - `tests/unit/ui/manager-tab-grid.test.ts`
  - `tests/e2e/manager-workflows.spec.ts`

## 1. 背景

当前 manager 链接卡片已经具备清晰的内容结构：

- 左侧图标区显示 favicon 或标题首字母
- 右侧文本区显示标题、hostname 和时间信息
- 整卡使用统一的浅暖色背景与固定文字体系

这套设计稳定，但当列表中卡片较多时，卡片之间的视觉层次主要依赖标题文本和留白，用户在快速扫读时较难建立“块状分区”的感知。用户希望通过颜色增强卡片层次，但明确要求：

- 优先提升扫读效率，而不是强化品牌感
- 卡片文字体系尽量保持统一，不做按卡片切换深浅字色
- 没有 favicon 的卡片也需要稳定、一致的颜色处理

## 2. 目标与非目标

### 2.1 目标

- 为 manager 卡片增加轻微而稳定的色相差异，提升长列表扫读时的分层感。
- 保持卡片整体仍然处于浅色、低饱和背景区间，避免牺牲文字可读性。
- 保持现有标题蓝、域名灰、时间辅助色不变。
- 让缺失 favicon 的卡片仍然具备稳定颜色，而不是退化成随机或失控样式。

### 2.2 非目标

- 不追求 favicon 品牌色的高保真还原。
- 不在首版中做 favicon 像素级取色或运行时 canvas 分析。
- 不为不同卡片切换不同文字颜色。
- 不引入深底卡或高饱和卡片。
- 不改变 manager 卡片的信息结构、交互行为或 action 布局。

## 3. 方案对比

### 3.1 方案 A：真实 favicon 主色轻染整卡

做法：从 favicon 中提取主色，直接把该颜色浅化后用于整卡背景。

优点：

- 与网站品牌的关系最直接。

缺点：

- 不同站点主色在明度和纯度上波动较大，即使浅化后也容易造成卡片节奏不稳定。
- 当 favicon 无法读取像素或跨域受限时，行为不稳定。
- 更容易让现有固定文字体系落入低对比区域。

### 3.2 方案 B：色相分桶 + 固定浅度轻染（推荐）

做法：不直接使用 favicon 原始颜色，而是根据稳定输入把卡片归入少数几个浅色家族，再用统一亮度和低饱和背景渲染整卡。

优点：

- 扫读稳定性最好，列表整体节奏一致。
- 颜色差异存在，但不至于造成视觉噪音。
- 不需要改动文字配色体系。
- 缺失 favicon 时也可以复用同一套映射规则。

缺点：

- 与具体网站品牌的颜色对应关系会更抽象。

### 3.3 方案 C：中性底色 + 少量彩色边缘

做法：保留当前中性卡片底色，只在边缘、顶部或 icon 区增加少量色彩。

优点：

- 风险最低，最不容易破坏对比度。

缺点：

- 对“整张卡片更有层次”的改善较弱。
- 用户想要的“整卡轻微染色”目标无法充分达成。

## 4. 推荐设计

### 4.1 总体策略

首版采用“色相分桶 + 固定浅度轻染”。

- 每张卡基于稳定输入得到一个色相家族。
- 所有家族都落入统一的浅色、低饱和区间。
- 色差只承担“增强层次”的职责，不承担品牌识别的职责。
- 文字颜色体系保持现状，不根据卡片背景切换。

### 4.2 颜色家族

首版建议使用少量稳定色系：

- `blue`
- `cyan`
- `green`
- `orange`
- `pink`
- `neutral`

设计原则：

- 家族数量要足够少，避免列表出现过细的颜色碎片化。
- 家族之间的亮度区间要尽量接近，差异主要来自轻微色相偏移，而不是明暗差。
- `neutral` 作为兜底色系，也可用于极低信息量或不可解析数据。

### 4.3 背景强度

用户已明确选择“非常轻”的染色强度，因此首版应遵守以下约束：

- 整卡背景仍然接近当前浅底风格，只在底色中加入轻微色相偏移。
- 不使用高饱和填充，不做明显色块对撞。
- 卡片之间的差异应在“一眼有层次，但不显花”的范围内。

### 4.4 图标区与占位字母

- 有 favicon 时，图标区继续显示 favicon。
- 没有 favicon 时，继续显示标题首字母占位。
- 图标容器和首字母占位块应与卡片使用同一家族颜色，但比卡片背景略深一档，以建立层次。
- 缺失 favicon 的卡片不应看起来像“异常状态”，而应看起来像同一视觉体系下的简化展示。

### 4.5 无 favicon 的颜色规则

首版不依赖 favicon 取色，因此“无 favicon”应当是正常场景，而不是退化路径。

规则如下：

- 若可从 `savedTab.url` 解析出 `hostname`，则基于 `hostname` 做稳定映射。
- 若 `hostname` 无法解析，则退回 `neutral` 家族。
- 同一 `hostname` 在不同会话、不同刷新、不同排序下都应映射到同一颜色家族。
- 不允许随机颜色，不允许刷新后色系漂移。

### 4.6 不做 favicon 像素取色

首版明确不做以下能力：

- 运行时将 favicon 绘制到 canvas 后做主色分析
- 因 favicon 异步取色而导致的首屏闪烁或颜色后补
- 因跨域限制导致的部分卡片可取色、部分卡片不可取色的不一致行为

这项能力如果后续需要，可作为二阶段增强，但增强后的结果也只应影响“落入哪个浅色家族”，而不是直接驱动整卡原色铺底。

## 5. 实现边界

### 5.1 结构边界

- 颜色分配逻辑不应直接散落在 React JSX 中。
- 建议抽出一个纯函数，例如根据 `url` 或 `hostname` 返回卡片色系标识。
- `ManagerTabGrid` 只负责把色系标识挂到 class 或 data attribute 上。
- 具体视觉表现由 `app-shell.css` 管理。

### 5.2 样式边界

- 可通过 `data-color-family` 或语义 class 控制卡片背景、icon 容器背景和占位字母颜色。
- 不改变当前标题、域名、时间文本的既有配色规则。
- hover、focus、dragging、drop-target 状态的反馈必须继续清晰，不应被新底色吞掉。

### 5.3 兼容当前密度模式

- `enhanced` 和 `compact` 密度都应共用同一套色系分配逻辑。
- 自动降级为紧凑密度时，色彩策略不变，只允许视觉强度自然减弱，不应出现另一套独立配色。

## 6. 测试与验收

### 6.1 单元测试

- 新增或更新纯函数测试，验证：
  - 相同 `hostname` 总是得到相同色系
  - 不同 `hostname` 可被稳定归入有限色系
  - 非法 URL 或不可解析输入回退到 `neutral`

### 6.2 组件测试

- `ManagerTabGrid` 测试应覆盖：
  - 有 favicon 卡片能渲染对应色系标识
  - 无 favicon 卡片仍能渲染稳定色系标识
  - 文本结构与 action 结构不因色系变化而改变

### 6.3 端到端关注点

- manager 长列表中，卡片在视觉上出现轻微分层，但文字仍清晰可读。
- hover/focus/drag/drop 状态不会因新底色而丢失辨识度。
- 固定分组、回收站、无 favicon 卡片在同一列表中仍保持一致视觉逻辑。

### 6.4 验收标准

- 列表整体视觉层次比现状更强，但不出现明显花哨感。
- 不需要修改现有文字颜色即可保持可读性。
- 同站点无 favicon 卡片不会随机变色或退化成无规律样式。
- 不引入 favicon 取色失败、异步闪烁或跨域异常带来的体验不一致。

## 7. 影响范围

- `src/ui/manager/ManagerTabGrid.tsx`
  - 为卡片和 icon 区挂接稳定色系标识。
- `src/ui/shared/app-shell.css`
  - 新增卡片色系样式、icon 区层次和无 favicon 占位表现。
- `tests/unit/ui/manager-tab-grid.test.ts`
  - 补充色系标识与无 favicon 展示的组件断言。
- 如需抽出纯函数：
  - 建议放在 `src/ui/manager/` 下的小型辅助模块或 `shared` 中的明确职责模块，并为其补单元测试。
