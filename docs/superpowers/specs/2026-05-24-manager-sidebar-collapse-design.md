# Manager Sidebar Collapse Design

- 适用范围：TabVault 管理页（Session Manager）左侧分组边栏折叠/展开能力
- 最后更新时间：2026-05-24
- 关联文件：
  - `src/ui/manager/App.tsx`
  - `src/ui/shared/app-shell.css`
  - `src/types/settings.ts`
  - `src/storage/local/schema.ts`
  - `docs/architecture/data-model.md`
  - `tests/integration/storage/repository.test.ts`
  - `tests/e2e/manager-workflows.spec.ts`

## 1. 背景

当前管理页左侧边栏承担分组浏览、选择和跨组拖拽目标承接的职责。此前已经通过 sticky 布局解决“右侧长列表滚动时左侧离开视口”的问题，但左侧边栏仍然始终占据一段固定宽度。

在用户主要关注右侧标签整理、搜索和批量浏览时，左侧边栏的持续占位会压缩右侧可用空间，尤其会影响增强密度卡片或长标题内容的可读性。用户已经明确希望增加“整栏折叠/展开”能力，并要求折叠状态在重新打开管理页后仍能保持。

## 2. 目标与非目标

### 2.1 目标

- 为管理页左侧边栏增加整栏折叠/展开控制。
- 首次进入管理页时默认展开。
- 折叠后保留一条窄栏，而不是完全隐藏边栏。
- 窄栏中保留明确的“展开边栏”入口，保证功能可发现。
- 用户手动切换后的边栏状态需要持久化保存，下次进入管理页时恢复上次状态。
- 保持现有分组选择、右侧详情展示、拖拽和搜索业务逻辑不变。

### 2.2 非目标

- 不重写左侧分组树的数据结构或排序规则。
- 不调整现有分组展开/收起行为（`全部`、`回收站` 两段树节点逻辑保持原样）。
- 不改变右侧标签区的信息密度偏好逻辑。
- 不在本次引入边栏拖拽调整宽度、自动折叠、悬浮展开或动画编排系统。
- 不修改存储 backend 选择逻辑，只在现有 settings 持久化模型中扩展字段。

## 3. 方案对比

### 3.1 方案 A：把折叠状态放进 `ExtensionSettings`（推荐）

做法：在现有 `RootState.settings` 中新增一个 manager 边栏偏好字段，管理页初始化时读取，用户切换时写回。

优点：

- 复用当前 settings 的默认值、migration、读写和跨页面同步链路。
- 与现有 `managerGridDensityPreference` 的产品语义一致，都是 manager 级别用户偏好。
- 状态来源单一，后续维护成本低。

缺点：

- 需要补 settings schema 默认值、迁移逻辑和相关测试。
- 需要同步更新文档和 ADR。

### 3.2 方案 B：为边栏折叠状态单独增加一个新的 local storage key

做法：绕开 `RootState.settings`，单独写入一个用于 manager sidebar 的浏览器本地 key。

优点：

- 表面上改动看起来更局部。

缺点：

- 会引入第二套设置存储入口，破坏当前统一的 settings 模型。
- 读取和同步路径会分叉，后续更容易出现状态不一致。
- 与项目“设置结构变更走 schema + migration”的既有约定不一致。

### 3.3 方案 C：只保留运行时 React state

做法：边栏折叠仅存在于 manager 页面内存状态，不写回存储。

优点：

- 实现最省。

缺点：

- 不满足“需要记住”的明确需求。

## 4. 推荐设计

### 4.1 交互设计

- 管理页左侧边栏在展开态时显示一个“折叠边栏”按钮。
- 该按钮应放在左侧边栏顶部的稳定位置，不随当前分组选中结果变化。
- 用户点击后，左侧完整分组树收起为一条固定宽度的窄栏。
- 折叠态窄栏中只保留一个明显的“展开边栏”按钮，不再渲染完整分组树内容。
- 用户点击“展开边栏”后，左侧恢复完整分组树展示。

### 4.2 默认状态与记忆规则

- 新用户或旧数据迁移后，默认状态为 `expanded`。
- 用户主动折叠后，后续再次进入管理页应保持 `collapsed`。
- 用户主动展开后，后续再次进入管理页应保持 `expanded`。
- 运行时因窗口缩放、右侧内容变化或数据刷新，不应自动改写用户偏好。

### 4.3 布局行为

- 展开态继续使用当前两栏工作台布局。
- 折叠态下，`.manager-workbench` 仍保持两栏结构，但左栏宽度收窄为固定 rail。
- 右侧 `.manager-main` 自动获得更多可用宽度，用于标签浏览、宫格展示和详情操作。
- 现有 sticky sidebar 行为在展开态继续保留。
- 折叠态 rail 本身也保持可见，不因为右侧长内容滚动而消失。

### 4.4 响应式策略

- 桌面宽度下，展开态和折叠态都沿用双栏结构。
- 当当前响应式规则把 manager 工作区回退为单列自然流布局时，折叠偏好仍然保留，但视觉上不再强求维持并排 rail。
- 在单列布局下，折叠态可以退化为仅显示“展开边栏”入口，再展开后恢复完整左侧分组树块。
- 该退化只影响表现方式，不影响用户保存的偏好值。

## 5. 数据与状态设计

### 5.1 设置字段

建议在 `src/types/settings.ts` 中新增如下类型和字段：

```ts
export type ManagerSidebarPreference = "expanded" | "collapsed";
```

```ts
managerSidebarPreference: ManagerSidebarPreference;
```

字段语义：

- `expanded` 表示左侧边栏完整展开。
- `collapsed` 表示左侧边栏折叠为窄栏。

### 5.2 默认值与 migration

- `defaultSettings` 中新增 `managerSidebarPreference: "expanded"`。
- `migrateRootState()` 在处理旧版本 `settings` 时，若不存在该字段，则补为 `"expanded"`。
- migration 只补默认值，不覆盖旧数据中已存在的其他 settings 字段。

### 5.3 Manager 页运行时状态

- `ManagerApp` 在初始化加载 settings 时，读取 `managerSidebarPreference` 并映射到本地 UI state。
- 用户点击折叠/展开时，界面先乐观更新，再异步调用 `saveSettings()` 持久化。
- 若持久化失败，恢复上一个 UI 状态，并通过现有状态消息区域告知失败原因。
- 当 `chrome.storage.onChanged` 触发 manager 页重新同步时，同步刷新边栏偏好，保证页面与存储状态一致。

## 6. UI 结构与样式约束

### 6.1 `src/ui/manager/App.tsx`

- 尽量保持现有左栏与右栏 DOM 层级不变。
- 在 `manager-workbench` 或 `manager-sidebar` 上增加可表达折叠态的 class 或 data attribute。
- 将边栏折叠/展开按钮纳入左栏顶部，避免把该控制散落到右侧主内容区。
- 折叠态下不渲染完整分组树内容，以减少窄栏中的布局噪声和焦点陷阱。

### 6.2 `src/ui/shared/app-shell.css`

- 为展开态与折叠态定义明确的 sidebar 宽度规则。
- 折叠态 rail 宽度应足够承载一个可点击、可聚焦的展开按钮。
- 折叠态不得把右侧主内容挤压成不可用窄列。
- 展开/折叠切换可以使用轻量过渡，但不将复杂动效作为本次验收前提。

### 6.3 可访问性

- 折叠/展开按钮必须有清晰可读的文本或 `aria-label`。
- 当前按钮状态应能被辅助技术识别，例如通过 `aria-expanded` 反映展开与折叠。
- 折叠后，完整分组树内容既然不再渲染，就不会残留不可见但仍可聚焦的元素。

## 7. 对现有行为的影响

以下行为在本次设计中保持不变：

- 分组选择逻辑
- `全部` 与 `回收站` 分区展开/收起逻辑
- 分组菜单、固定、重命名、移入回收站、恢复、永久删除等能力
- 右侧标签展示、搜索命中聚焦、标签拖拽与跨组移动
- manager grid density 偏好的语义与持久化方式

折叠态只改变左栏占用空间和内容可见性，不应改变业务结果。

## 8. 异常与边界处理

- 如果保存边栏偏好失败，界面应回滚到上一个状态，避免“看起来折叠了但刷新后丢失”的假成功。
- 如果读取到的旧数据没有该字段，页面应稳定按 `expanded` 启动，而不是出现空值分支。
- 如果 manager 页在折叠态下接收到 storage 同步更新，不应强制把边栏重新展开，除非最新 settings 明确写的是 `expanded`。
- 如果用户在窄屏单列布局下已经保存为 `collapsed`，再次进入时仍应看到可操作的展开入口，而不是完全失去左栏访问方式。

## 9. 验收标准

- 首次进入管理页时，左侧边栏默认展开。
- 点击折叠按钮后，左侧边栏收为窄栏，且窄栏中存在明确的“展开边栏”入口。
- 点击展开入口后，左侧完整分组树重新显示。
- 折叠后刷新管理页，边栏仍保持折叠。
- 展开后刷新管理页，边栏仍保持展开。
- 折叠/展开切换不影响现有分组选择、搜索、拖拽和右侧内容浏览。

## 10. 测试策略

### 10.1 单元与集成测试

- `repository` 测试补充：
  - 默认 settings 含 `managerSidebarPreference: "expanded"`
  - 更新 settings 后可以持久化 `collapsed`
- `schema` / migration 测试补充：
  - 旧 settings 缺失该字段时自动补默认值
- UI / 样式测试补充：
  - 折叠态相关样式选择器、宽度规则或状态类存在

### 10.2 E2E 测试

- 打开 manager 页，确认初始为展开态。
- 点击折叠按钮，确认左栏进入窄栏态，展开入口可见。
- 刷新页面，确认仍保持折叠态。
- 再点击展开入口，确认完整分组树恢复。
- 在展开态与折叠态之间切换后，右侧主内容仍可正常浏览。
