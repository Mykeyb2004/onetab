# Manager Sticky Sidebar Design

- 适用范围：TabVault 管理页（Session Manager）左右两栏拖拽体验优化
- 最后更新时间：2026-04-21
- 关联文件：`src/ui/manager/App.tsx`、`src/ui/shared/app-shell.css`

## 1. 背景

当前管理页采用左右两栏布局：左侧展示分组树，右侧展示选中分组详情与标签列表。右侧标签较多时，页面会整体向下滚动，导致用户在拖拽右侧标签到左侧其他分组时，左侧分组列表可能已经离开可视区，跨组移动操作变得困难。

本次优化目标是：在不改变现有拖拽模型的前提下，让左侧分组列表在拖拽期间和普通浏览期间都尽量保持可见，从而降低跨组移动的操作成本。

## 2. 目标与非目标

### 2.1 目标

- 左侧分组列表在桌面宽度下保持可见，不会随着右侧长列表一起滚出视口。
- 右侧分组详情和标签列表支持独立滚动。
- 保持现有拖拽行为、数据流和业务逻辑不变，优先通过布局优化解决问题。
- 保持移动端或窄屏下的可用性，避免 sticky 布局挤压内容。

### 2.2 非目标

- 不重写现有标签拖拽逻辑。
- 不新增浮动投放面板、自动滚屏或全新跨组移动交互。
- 不调整分组数据结构、持久化模型或消息协议。

## 3. 方案对比

### 3.1 方案 A：左侧 sticky，右侧独立滚动（推荐）

做法：左侧分组栏使用 sticky 定位，右侧详情区使用独立滚动容器，整页不再因为右侧长列表而把左栏一起卷走。

优点：

- 直接解决“拖拽时看不到左侧目标分组”的根因。
- 改动范围主要集中在管理页布局和样式，风险较低。
- 不改变用户已习惯的拖拽路径和心智模型。

缺点：

- 需要小心处理视口高度、顶部间距和窄屏降级。

### 3.2 方案 B：整页滚动但左侧完全固定

做法：左侧作为固定栏脱离普通文档流，右侧继续整页滚动。

优点：

- 视觉上左栏始终稳定。

缺点：

- 与当前布局耦合更强，容易引入覆盖、遮挡或高度计算问题。
- 顶部内容与主内容的滚动关系会更难维护。

### 3.3 方案 C：拖拽时弹出目标分组面板

做法：拖拽标签时显示一个额外的浮动面板供用户直接投放。

优点：

- 拖拽路径最短。

缺点：

- 超出本次“最小改动优化体验”的范围。
- 需要新增较多状态、可访问性处理和测试。

## 4. 推荐设计

### 4.1 布局

- 保留现有 `.manager-workbench` 两栏结构。
- 在桌面宽度下，为工作区设置接近视口高度的可视区域。
- 左侧 `.manager-sidebar` 使用 `position: sticky`，吸附在页面顶部安全间距之下。
- 右侧 `.manager-main` 作为独立滚动区域承载详情卡片和标签列表。

### 4.2 交互

- 用户向下浏览右侧长标签列表时，左侧分组树仍保持在当前视口中。
- 用户开始拖拽标签时，不需要先滚回顶部即可看到左侧目标分组。
- 现有分组高亮、拖拽目标高亮、放置逻辑保持不变。

### 4.3 响应式降级

- 在窄屏断点下，取消 sticky 与独立滚动，恢复为纵向自然流布局。
- 避免在小屏幕上出现两个并列滚动区域，降低操作复杂度。

## 5. 影响范围

### 5.1 `src/ui/shared/app-shell.css`

- 调整 `.manager-workbench` 的高度与对齐方式。
- 为 `.manager-sidebar` 增加 sticky 定位与最大高度约束。
- 为 `.manager-main` 增加独立滚动容器样式。
- 在移动端断点下显式关闭 sticky / 高度限制。

### 5.2 `src/ui/manager/App.tsx`

- 预计无需改动业务逻辑。
- 仅当现有 DOM 层级不利于独立滚动时，才补充最小结构包裹层或类名。

## 6. 数据流与状态

本次优化只涉及布局和滚动容器，不改变以下内容：

- 标签拖拽数据载荷（`TAB_DRAG_TYPE`）
- 分组拖拽数据载荷（`SESSION_DRAG_TYPE`）
- `dragOverSessionId`、`dragOverTabId`、`draggedTabId` 等界面状态
- 分组读写、跨组移动、排序和持久化流程

因此该改动属于纯 UI 壳层优化，不应影响业务结果。

## 7. 异常与边界处理

- 若右侧内容较短，右侧滚动容器应保持正常显示，不出现多余空白滚动条。
- 若左侧分组数量很多，左侧自身也应允许内部滚动，但仍保持 sticky 在视口内。
- 若浏览器窗口高度较小，sticky 顶部偏移与最大高度应避免遮挡顶部头部区域。
- 若进入窄屏断点，必须回退到单列自然滚动，避免双滚动区域难以操作。

## 8. 验收标准

- 当右侧标签列表足够长时，滚动右侧内容不会让左侧分组栏离开视口。
- 用户从右侧任一可见标签开始拖拽时，左侧分组列表可见且可直接投放。
- 现有标签重排、跨组移动、分组选中和更多操作按钮行为不变。
- 窄屏下布局不出现遮挡、截断或不可滚动问题。

## 9. 测试策略

### 9.1 手工验证

- 在桌面宽度下构造长标签列表，确认右侧滚动时左侧仍可见。
- 从右侧靠下位置拖拽标签到左侧其他分组，确认投放可完成。
- 验证左侧分组很多时，左侧内部滚动仍可用。
- 缩小窗口到移动端断点附近，确认布局回退正常。

### 9.2 自动化验证

- 单元测试：无需新增，因无业务逻辑变化。
- 集成测试：无需新增，因无数据流变化。
- 端到端测试：可补一个管理页长列表拖拽场景，验证左侧分组栏在长列表下仍可作为投放目标。

## 10. 实施顺序

1. 先调整管理页容器高度与左右栏滚动关系。
2. 再为左侧侧栏增加 sticky 和内部滚动约束。
3. 最后验证桌面与窄屏下的拖拽和浏览体验。
