# Manager Grid Density View Design

- 适用范围：TabVault 管理页（Session Manager）右侧标签区视图切换重构
- 最后更新时间：2026-05-24
- 关联文件：
  - `src/ui/manager/App.tsx`
  - `src/types/settings.ts`
  - `src/storage/local/schema.ts`
  - `docs/architecture/data-model.md`
  - `tests/e2e/manager-workflows.spec.ts`

## 1. 背景

当前管理页右侧标签区采用单列列表展示，每条记录同时承载标题、URL 和多个文字按钮。这个结构的优点是操作直达，但也带来几个明显问题：

- 长分组下单屏可见标签数量偏少，滚动成本高。
- 文字按钮占据较大面积，内容识别层被挤压。
- 当用户主要目标是扫读、拖拽、整理而不是逐条执行操作时，界面显得过重。

本次讨论后，用户已明确不再需要“列表视图 / 宫格视图”的多主视图体系，而是希望把右侧标签区统一改为一种自适应宫格视图，再在宫格内部支持两档信息密度切换。

## 2. 目标与非目标

### 2.1 目标

- 右侧标签区统一改为响应式宫格，让一屏可见更多标签。
- 视图切换定义为宫格内部的两档信息密度切换：`简洁` 与 `增强`。
- 操作按钮从重文字按钮改为悬浮小图标，降低版面负担。
- 保留现有组内拖拽、跨组移动、恢复、删除等核心能力。
- 在窄宽度下自动退化到可读性更高的简洁密度，但不丢失用户手动偏好。
- 将偏好持久化，用户重新打开 manager 后可恢复上次选择。

### 2.2 非目标

- 不保留第二种主视图，例如识别增强列表视图。
- 不改变底层标签顺序模型、恢复顺序或导出语义。
- 不重写左侧分组树或跨组拖拽模型。
- 不在本次引入批量选择、批量操作或新的搜索模式。
- 不修改回收站业务规则，回收站中的标签仍然主要随分组一起恢复。

## 3. 方案对比

### 3.1 方案 A：多主视图并存

做法：保留多个主视图，例如宫格视图加识别增强列表视图。

优点：

- 可覆盖更多使用偏好。
- 列表视图更容易延续当前操作布局。

缺点：

- 用户已明确不需要多主视图。
- 状态、测试和文案复杂度更高。
- 会把“信息密度切换”问题重新放大成“主视图选择”问题。

### 3.2 方案 B：单一宫格 + 完全自动密度适配

做法：界面根据宽度自动在简洁和增强之间切换，不提供显式控制。

优点：

- 控件更少，界面更干净。
- 响应式逻辑集中。

缺点：

- 用户无法稳定表达个人偏好。
- 在边界宽度附近容易出现“信息忽多忽少”的不确定感。

### 3.3 方案 C：单一宫格 + 手动密度切换 + 自动降级（推荐）

做法：只保留一种响应式宫格视图，用户手动选择 `简洁 / 增强`，当宽度不足以承载增强密度时，界面临时自动降为简洁；宽度恢复后自动回到用户原偏好。

优点：

- 同时满足可控性与响应式稳定性。
- 心智简单：只有一种主视图，切换的是信息密度。
- 易于和现有拖拽、恢复逻辑共存。

缺点：

- 需要同时维护“用户偏好密度”和“当前实际生效密度”两个概念。
- 自动降级规则必须清晰且可测试。

## 4. 推荐设计

### 4.1 信息架构

- 右侧标签区只有一种主视图：响应式宫格。
- 所谓“视图切换”在产品层面重新定义为信息密度切换，而不是布局切换。
- 两档密度分别命名为：
  - `简洁`
  - `增强`

### 4.2 切换控件

- 在当前分组详情头部提供显式切换控件。
- 控件与 `全部还原`、`更多操作` 同属右侧详情头部区域，只作用于当前右侧标签区。
- 文案直接使用 `简洁 / 增强`，避免再使用会引起歧义的“视图一 / 视图二”命名。

### 4.3 宫格布局

- 标签以卡片形式呈现。
- 宫格列数基于右侧可用宽度自动计算，形成用户所需的 `n x n` 自适应布局。
- 宫格只改变呈现方式，不改变底层顺序模型。
- 真实顺序仍然是线性序列，视觉上映射为从左到右、从上到下。

### 4.4 卡片内容

#### 简洁模式

- `favicon`
- 标签标题
- 主域名

#### 增强模式

- 包含简洁模式的全部信息
- 再增加一行轻量 meta
- 允许使用的 meta 类型包括：
  - 站点标签或分类
  - 相对时间信息
  - 来源或轻量辅助说明

约束：

- 增强模式只增加一行轻量信息，不回退到重列表卡片。
- 标题仍然是卡片识别主轴，meta 不得压过标题层级。

### 4.5 悬浮小图标操作

- `打开`、`还原并移除`、`删除` 统一改为卡片右上角的小图标操作。
- 图标位置固定，不因 hover 或 focus 改变布局。
- 鼠标悬停卡片时，图标淡入显示。
- 键盘聚焦卡片时，也要显示同样的图标和可见的 focus 高亮。
- 继续通过键盘导航时，应可进入具体图标按钮并触发操作。
- 在触控或无 hover 场景下，首次点按卡片只进入选中态并露出图标，避免误触直接执行动作。
- 图标区域不参与拖拽，正文区域保留拖拽入口。

### 4.6 拖拽与排序

- 组内拖拽能力保持。
- 宫格中的落点以卡片占位插槽表达，而不是列表中的横向插入线。
- 拖到左侧分组树时，继续沿用当前跨组移动逻辑，不引入新的跨组交互模型。
- 回收站中的卡片不支持单卡片拖拽，也不显示这组悬浮操作图标。

### 4.7 回收站行为

- 回收站中仍使用同一套宫格和密度逻辑，保持视觉一致。
- 由于回收站标签页当前不支持逐条恢复与删除的常规流，卡片只承担展示作用，不显示活跃操作图标。
- 回收站的分组级恢复和清空操作继续停留在分组详情头部。

### 4.8 响应式与自动降级

- 用户可手动选择 `简洁` 或 `增强` 作为偏好。
- 当右侧可用宽度不足以让增强卡片保持可读性时，界面临时强制使用 `简洁`。
- 一旦宽度恢复，界面自动回到用户之前选择的偏好密度。
- 自动降级依据应基于卡片最小可读宽度和容器宽度，而不是单纯依赖固定窗口断点。

## 5. 数据与状态设计

### 5.1 持久化设置

实现阶段需要在 `ExtensionSettings` 中新增一个轻量字段用于记录用户偏好密度。

建议语义：

```ts
type ManagerGridDensityPreference = "compact" | "enhanced";
```

建议字段：

```ts
managerGridDensityPreference: ManagerGridDensityPreference;
```

默认值建议为 `enhanced`，理由如下：

- 首次进入新宫格视图时，能更完整地展示该视图价值。
- 当宽度不足时，系统仍会自动退化到简洁模式，不影响可读性。

### 5.2 运行时派生状态

除持久化偏好外，界面还需要一个运行时派生结果，例如“当前实际生效密度”。

其计算原则为：

1. 先读取用户偏好。
2. 再根据当前右侧可用宽度判断增强模式是否可用。
3. 若不可用，则临时生效为 `简洁`。
4. 若可用，则恢复为用户偏好。

### 5.3 Migration 要求

- 老用户缺失该字段时，必须补默认值，不得覆盖已有其他设置。
- 本次属于 `settings` 结构扩展，读取旧 root state 时必须经过兼容升级。
- 自动降级状态属于运行时状态，不写回持久化设置。

## 7. 验收标准

- 管理页右侧标签区默认以响应式宫格展示，不再保留列表型主视图。
- 用户可显式切换 `简洁 / 增强` 两档密度。
- 宫格会随右侧可用宽度自适应列数。
- 在宽度不足时，增强模式会临时退化到简洁模式；宽度恢复后会自动回到用户偏好。
- 卡片的 `打开 / 还原并移除 / 删除` 以悬浮小图标呈现，不再使用重文字按钮。
- hover、keyboard focus 和触控场景下都能可靠触达卡片操作。
- 组内拖拽和跨组移动保持可用，恢复顺序和持久化顺序语义不变。
- 回收站中的卡片保持宫格展示，但不显示单卡片活跃操作图标。

## 8. 测试策略

### 8.1 单元测试

- 密度判定规则：用户偏好增强但宽度不足时，应生效为简洁。
- 恢复规则：宽度恢复后，应回到用户原偏好。
- 若存在卡片 meta 选择逻辑，应验证其在简洁和增强模式下的输出差异。

### 8.2 集成测试

- 新增设置字段的默认值补齐。
- 旧数据 migration 后不丢失已有 settings 字段。
- 自动降级为运行时状态，不写回持久化数据。

### 8.3 端到端测试

- 用户切换 `简洁 / 增强` 后刷新页面，偏好仍然保留。
- 缩小右侧宽度时自动降为简洁，恢复宽度后回到增强。
- hover 或 focus 卡片时操作图标出现。
- 点击图标不会误触发拖拽。
- 宫格下组内拖拽仍然有效。
- 宫格下拖到左侧分组树的跨组移动仍然有效。

## 9. 实施顺序

1. 先扩展设置模型与 migration，确保偏好可安全持久化。
2. 再抽离宫格密度判定与运行时降级逻辑。
3. 之后重构 manager 右侧标签区为统一宫格卡片。
4. 再补充悬浮小图标、键盘和触控交互细节。
5. 最后补齐单元、集成和端到端测试，并同步更新数据模型文档与 ADR。
