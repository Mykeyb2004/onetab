# OneTab 类 Chrome 插件 PRD

## 1. 文档信息

- 文档版本：v1.1
- 文档日期：2026-04-19
- 产品代号：TabVault
- 产品形态：Chrome Extension（Manifest V3）
- 产品定位：将分散的浏览器标签页快速收纳为可恢复、可搜索、可整理的会话列表，降低标签混乱与内存占用。

## 2. 产品背景

重度浏览器用户往往同时打开几十到上百个标签页，常见问题包括：

- 标签栏过载，难以定位重要页面
- 浏览器内存占用高，影响系统性能
- 工作上下文分散，切换成本高
- 用户想暂存一批页面，但又不想永久加入书签

OneTab 验证了“标签收纳为列表”的核心需求。本产品将提供一个相似但更现代、可扩展、默认更注重隐私的 Chrome 插件。

## 3. 产品目标

### 3.1 核心目标

1. 让用户能够一键收纳当前标签页，减少视觉噪音与资源占用。
2. 让用户能够随时按组或按单个标签恢复原有工作上下文。
3. 让已收纳标签具备基础管理能力，包括命名、搜索、删除、批量恢复与导入导出。
4. 全流程以本地优先为原则，不依赖服务端即可使用。

### 3.2 成功标准

- 新用户首次安装后 1 分钟内完成第一次标签收纳
- 收纳 50 个标签页的操作耗时小于 2 秒
- 恢复单个标签页成功率大于 99.5%
- 7 日内至少有 30% 用户进行第二次收纳

### 3.3 非目标

- 不做传统书签管理器替代品
- 不做跨浏览器同步平台的首版能力
- 不做团队协作知识库
- 不做标签内容抓取、网页全文索引或云端备份的首版能力

## 4. 目标用户

### 4.1 核心用户

- 开发者、设计师、研究人员、运营、学生
- 每天打开大量标签页并在多个任务间切换的人
- 需要临时保存一组网页，但不希望污染书签栏的人

### 4.2 用户画像

- 用户 A：资料收集型
  - 打开大量文章、文档、视频和 GitHub 页面
  - 需要按课题或任务暂存一组链接
- 用户 B：多项目切换型
  - 不同客户或项目对应不同窗口与标签组
  - 需要快速关闭当前上下文，并在之后完整恢复
- 用户 C：性能敏感型
  - 机器内存有限
  - 需要快速降低浏览器资源占用

## 5. 核心使用场景

1. 用户准备下班或切换任务，希望把当前窗口的全部标签收起来，稍后再恢复。
2. 用户做资料搜集时临时打开大量页面，希望按“选题”形成一个待办列表。
3. 用户浏览器卡顿，希望批量关闭标签但保留链接，不丢失上下文。
4. 用户想把一组已保存链接导出，迁移到另一台电脑或分享给他人。

## 6. 产品方案概述

产品由三个核心界面组成：

- 浏览器工具栏图标：承载高频一键操作
- Popup 弹窗：提供快捷操作入口
- Session Manager 页面：提供完整的收纳列表管理能力

默认产品逻辑：

- 用户点击插件图标后，看到快速操作弹窗
- 用户可一键收纳当前窗口、当前标签页或选中的标签页
- 收纳后生成一个“会话组”
- 会话组内保存标签标题、URL、创建时间、来源窗口信息
- 用户可在管理页中搜索、重命名、恢复、删除、导入、导出
- 会话组默认命名规则为 `保存于 YYYY-MM-DD HH:mm`
- 恢复整组标签时，默认新建窗口并按原始顺序恢复
- 恢复单个标签后，该标签会从原组移除
- 搜索范围包含组名、标签标题与 URL
- 纯文本导入遇到非法 URL 时跳过继续，并提示跳过数量
- 导出支持“导出全部”和“仅导出单组”
- MVP 不支持无痕窗口标签收纳

## 7. 功能范围

## 7.1 MVP 功能

### F1. 一键收纳当前窗口全部标签

- 入口：
  - Popup 主按钮
  - 右键菜单
  - 快捷键
- 行为：
  - 读取当前窗口全部标签
  - 过滤插件自身页面与不支持保存的特殊页面
  - 创建新会话组
  - 关闭已收纳的标签页
- 结果：
  - 成功提示用户已保存数量
  - 提供“立即查看”入口

### F2. 收纳当前标签页或选中标签页

- 支持只收纳当前活动标签页
- 支持收纳用户多选的标签页
- 未被选中的标签页保持打开

### F3. 会话组列表管理

- 按时间倒序展示会话组
- 每组显示：
  - 组名
  - 标签数量
  - 创建时间
  - 前 3 个标签标题预览
- 支持展开查看组内全部标签

### F4. 恢复能力

- 支持恢复单个标签页
- 支持恢复整组标签页
- 恢复整组标签时，默认在新建窗口中按原始顺序打开
- 恢复单个标签后，默认从原组移除该标签
- 支持“恢复后保留原组”与“恢复后移除原组”两种策略
- 默认策略：恢复后移除原组，减少重复堆积

### F5. 组管理

- 支持会话组重命名
- 支持删除单个标签
- 支持删除整组
- 支持手动固定重要组到顶部，并在页面右键菜单中作为长期发送目标

### F6. 搜索

- 按组名、标签标题与 URL 搜索已保存内容
- 搜索结果可直接打开目标标签，或定位到所属组

### F7. 导入与导出

- 支持导出全部会话组
- 支持仅导出单个会话组
- 导出格式：
  - 纯文本 URL 列表
  - JSON 备份
- 导入格式：
  - 纯文本 URL 列表
  - JSON 备份
  - SPD JSON（按分类导入为多个分组）
- 纯文本导入遇到非法 URL 时跳过继续，并提示跳过数量
- 任意导入格式遇到已存在于固定分组中的 URL 时应跳过该 URL，避免把固定收藏重复导入

### F8. 设置项

- 恢复后是否自动删除原组
- 点击插件图标后的默认行为
- 是否展示收纳成功通知
- 是否开启右键菜单

### F9. 本地存储与隐私

- 默认仅使用本地存储
- 不上传用户标签 URL、标题或分组数据到服务端
- 在设置页明确说明数据存储位置与隐私边界
- MVP 不支持无痕窗口标签收纳

## 7.2 V1.1 功能

- 拖拽调整组顺序
- 拖拽调整组内标签顺序
- 多选标签做批量删除或批量恢复
- 地址栏关键字搜索已保存标签
- 去重提示：发现相同 URL 时提醒用户合并或跳过

## 7.3 V2 候选功能

- 分享为只读页面
- 加密云备份
- 跨设备同步
- 自动规则收纳
  - 例如按域名、窗口、工作模式自动归档
- Side Panel 模式
- AI 辅助命名会话组

## 8. 关键交互流程

### 8.1 首次使用流程

1. 用户安装插件
2. 点击插件图标
3. Popup 展示三个主要按钮：
   - 收纳当前窗口
   - 收纳当前标签
   - 打开管理页
4. 用户执行首次收纳
5. 插件提示“已收纳 X 个标签页”
6. 用户点击“查看已收纳标签”进入管理页

### 8.2 收纳当前窗口流程

1. 用户点击“收纳当前窗口”
2. 系统获取当前窗口标签列表
3. 系统过滤不可收纳标签
4. 系统创建会话组并落盘
5. 系统关闭已收纳标签
6. Popup 或通知区返回操作结果

### 8.3 恢复整组流程

1. 用户进入管理页
2. 找到目标会话组
3. 点击“恢复全部”
4. 系统新建浏览器窗口，并按原始顺序打开标签
5. 若设置为“恢复后删除”，则移除该组

### 8.4 导出备份流程

1. 用户进入设置页或组操作菜单
2. 选择“导出全部”或“导出单组”，并选择导出格式
3. 系统生成文件并触发下载
4. 用户可在另一浏览器实例中导入

## 9. 详细需求

### 9.1 数据模型

#### SessionGroup

- `id`
- `title`（默认格式：`保存于 YYYY-MM-DD HH:mm`）
- `createdAt`
- `updatedAt`
- `tabCount`
- `pinned`
- `sourceWindowId`
- `tabs[]`

#### SavedTab

- `id`
- `title`
- `url`
- `favIconUrl`
- `createdAt`
- `lastOpenedAt`
- `originalIndex`

### 9.2 页面与信息架构

#### A. Popup

- 顶部：今日已收纳数量或当前总组数
- 主按钮区：
  - 收纳当前窗口
  - 收纳当前标签
  - 收纳选中标签
- 次级入口：
  - 打开管理页
  - 设置

#### B. Session Manager

- 顶部：
  - 搜索框
  - 导入
  - 导出
  - 设置
- 内容区：
  - 会话组列表
  - 组卡片
  - 组内标签列表
- 每个组的操作：
  - 恢复全部
  - 重命名
  - 固定
  - 导出本组
  - 删除

#### C. 设置页

- 恢复策略
- 默认点击动作
- 右键菜单开关
- 数据导入导出
- 隐私说明
- MVP 范围说明（含无痕窗口暂不支持）

### 9.3 异常与边界情况

- `chrome://`、`chrome-extension://`、新标签页等特殊页面不可正常恢复时，需要在 UI 中提示“部分页面未保存”
- URL 为空或标签尚未加载完成时，跳过并计数
- 收纳过程中如关闭标签失败，不影响其他标签保存
- 恢复大量标签时，需避免瞬时打开过多导致浏览器卡顿
  - 可采用分批打开策略
- 导入的 JSON 若版本不兼容，应提示并拒绝导入
- 纯文本导入中的非法 URL 应跳过处理，并反馈跳过数量
- MVP 不支持无痕窗口标签收纳；若用户进入相关说明页，应明确提示当前版本不支持

## 10. 非功能要求

### 10.1 性能

- 收纳 100 个标签时，主线程不可出现明显卡死
- 管理页首次打开时间小于 1.5 秒
- 搜索 5000 条已保存标签时，结果返回时间小于 200ms

### 10.2 隐私与安全

- 默认离线可用
- 默认不依赖远程服务器
- 不采集浏览历史以外的额外内容
- 不读取网页正文或表单数据

### 10.3 兼容性

- 支持 Chrome 最新稳定版及近两个大版本
- 适配 1280px 以上桌面分辨率
- Popup 在窄宽度下仍可完整操作

## 11. 技术约束与建议

### 11.1 Chrome 权限建议

- `tabs`
- `storage`
- `contextMenus`
- `notifications`
- `commands`
- 可选：`downloads`

### 11.2 Manifest V3 架构建议

- `service_worker`
  - 处理标签收纳、快捷键、右键菜单、恢复逻辑
- `action.popup`
  - 承载高频入口
- `options_page` 或独立管理页
  - 承载完整会话管理与设置

### 11.3 存储建议

- 首选 `chrome.storage.local`
- 当数据量增大时，可逐步迁移到 `IndexedDB`
- 导出 JSON 时带 `schemaVersion`

## 12. 埋点与指标

建议仅在用户明确同意后启用匿名埋点。若不做埋点，至少保留本地统计用于产品判断。

建议关注指标：

- 安装到首次收纳转化率
- 单次收纳标签数
- 恢复单个标签次数
- 恢复整组次数
- 搜索使用率
- 导入导出使用率
- 7 日留存与 30 日留存

## 13. 风险与对策

- 风险：用户担心数据丢失
  - 对策：增加导出备份、操作提示、恢复前确认
- 风险：一次恢复太多标签导致浏览器卡顿
  - 对策：分批恢复并提供进度反馈
- 风险：特殊页面无法保存引发误解
  - 对策：明确列出跳过数量与原因
- 风险：本地存储容量增长过快
  - 对策：按组压缩字段、清理冗余 favicon、必要时迁移 IndexedDB

## 16. 验收标准

1. 用户可在 3 次点击内完成“收纳当前窗口”。
2. 收纳后，管理页可看到完整保存的标签列表与数量。
3. 用户可成功恢复单个标签与整组标签。
4. 用户关闭浏览器后再次打开，已收纳数据仍存在。
5. 导出的 JSON 可被同版本插件重新导入并正确恢复。
6. 特殊页面无法保存时，系统会明确提示跳过数量。

## 17. 结论

该产品的 MVP 应聚焦于“快收纳、能恢复、好查找、不丢数据”四件事。只要把这四个基础动作做好，就已经具备替代 OneTab 核心价值的能力。后续差异化可以围绕更强的组织能力、更好的隐私体验与更现代的交互展开。
